# ТЗ — Қисми 2: Frontend (Flutter — Windows desktop)

> Техническое задание барои барномаи мизоҷ (касса/анбор). Дастур аз рӯи ин қадам ба қадам кор мекунад.
> Алоқа: `TZ_00_TAHLILI_ILOVAGI.md`, `TZ_01_BACKEND_DOTNET.md` (API-и ҳамин ҷо истеъмол мешавад).

**Стек:** Flutter (Windows desktop аввал) • Dart • Riverpod • Dio • go_router • Drift (offline)
**Платформа:** Windows 10/11 (баъд Android/Web имконпазир)

## 1. Архитектура (Feature-first + Clean)

```
lib/
├── main.dart
├── app/                      # MaterialApp, theme, router, DI (ProviderScope)
│   ├── app.dart
│   ├── router.dart           # go_router
│   └── theme.dart            # Material 3
├── core/                     # умумӣ барои ҳама
│   ├── api/                  # Dio client, interceptor (JWT, refresh, error)
│   ├── storage/              # secure storage (токен), Drift (offline DB)
│   ├── error/                # Failure, Exception
│   ├── constants/            # API URL, рангҳо, текст
│   └── utils/                # формати пул/сана, валидаторҳо
├── features/
│   ├── auth/                 # login
│   ├── pos/                  # касса (фурӯш)  ← модули асосӣ
│   ├── products/             # справочники дору
│   ├── receipts/             # приход
│   ├── stock/                # анбор, мӯҳлат
│   ├── operations/           # списание, инвентаризатсия, transfer
│   ├── reports/              # ҳисоботҳо
│   └── settings/             # корбарон, танзимот
└── shared/                   # widget-ҳои умумӣ (button, table, dialog)
```

**Ҳар feature:** `data/` (repository, model) → `domain/` (entity, usecase, ихтиёрӣ) → `presentation/` (screen, widget, provider).

## 3. Экранҳо (Screens) ва вазифаашон

### 3.1. Auth
- **Login** — логин/парол → токенро дар secure storage, → ба касса.

### 3.2. POS / Касса (МУҲИМТАРИН)
- **Кушодани смена** — агар смена кушода набошад, аввал openingCash.
- **Экрани фурӯш:**
  - майдони скан/ҷустуҷӯ (фокус ҳамеша ин ҷо);
  - ҷадвали чек (ном, миқдор, нарх, маблағ, тугмаи ±/ҳазф);
  - ҷамъи умумӣ калон дар поён;
  - тахфиф, тарзи пардохт (нақд/корт), "Пардохт" (F-tugma);
  - баъди пардохт → чоп + чеки нав.
- **Бозгашти фурӯш** — ҷустуҷӯи чек → интихоби сатр → возврат.
- **Бастани смена** — closingCash → Z-отчёт → чоп.
- **Hotkeys** (касса бо клавиатура): `F2` ҷустуҷӯ, `F4` тахфиф, `F9` пардохт, `Del` ҳазф.

### 3.3. Справочники дору (Products)
- Рӯйхат бо ҷустуҷӯ/филтр (гурӯҳ, истеҳсолкунанда).
- Корти дору: ном, штрих-код, гурӯҳ, ретсептӣ, воҳид, нархҳо.
- Сохтан/таҳрир (форма бо валидатсия).

### 3.4. Приход (Receipts)
- Рӯйхати накладнаҳо (сана, таъминкунанда, статус).
- Эҷоди приход: интихоби таъминкунанда → илова сатр (скан/ҷустуҷӯ) →
  миқдор, **серия**, **мӯҳлат**, нархи харид/фурӯш → "Тасдиқ (Provesti)".

### 3.5. Анбор (Stock)
- Бақия аз рӯи филиал (ном, миқдор, мӯҳлат, нарх).
- Таб "Мӯҳлати наздик" (90/30 рӯз) бо ранги огоҳӣ.
- Таб "Камшуда" (зери минимум).
- Таърихи ҳаракати дору.

### 3.6. Амалиёт (Operations)
- Списание (сабаб + сатрҳо), Инвентаризатсия (миқдори воқеӣ), Transfer.

### 3.7. Ҳисоботҳо (Reports)
- Фурӯш (давра, филиал, фурӯшанда), фоида, бақия, Z-отчёт.
- Чоп/экспорт (PDF/Excel).

### 3.8. Танзимот (Settings)
- Корбарон (Admin), нақшҳо, наценка, рӯзҳои огоҳӣ, принтер.
- (1 филиал дар оғоз → идоракунии филиал ва интихоби филиал ҳозир лозим нест.)

## 4. Кор бо API (муҳим)
- `Dio` бо `baseUrl` аз конфиг. Interceptor:
  - токенро ба header `Authorization: Bearer ...` мегузорад;
  - агар `401` → бо refresh-token нав мекунад, баъд такрор;
  - хатоҳоро ба `Failure` табдил медиҳад (барои UI).
- Ҳар feature → `Repository` (interface) → `RemoteDataSource` (Dio) +
  `LocalDataSource` (Drift, барои offline).
- Model-ҳо аз JSON-и Swagger мутобиқ — `freezed` + `json_serializable`.

---

## 5. Стратегияи Offline — ✅ ҲАТМӢ (қарори ТЗ_00)
> Қарор шуд: касса бояд бе интернет фурӯшад. Drift аз қадами POS ворид мешавад (на дилхоҳ).
1. **PULL:** справочники (доруҳо, нархҳо, бақия) дар Drift кэш — аз `GET /sync/catalog?since=` (delta).
2. Ҳангоми қатъи интернет: фурӯш дар Drift ҳамчун **pending** бо `ClientId` (Guid) сабт мешавад.
3. Чек чоп мешавад (offline ҳам — чеки оддӣ ESC/POS, бе фискалӣ).
4. **PUSH:** интернет баргашт → sync queue фурӯшҳоро `POST /sync/sales` мефиристад (idempotent бо `ClientId`).
5. Ҷавоб `ok` → аз pending тоза мешавад; `conflict` (бақия нарасид) → ба корбар огоҳӣ.
6. Конфликт: сервер манбаи ҳақиқат; бақия аз сервер аз нав кашида мешавад.

> **Эзоҳ:** бе интернет нархи кэшшуда истифода мешавад; FEFO-ро ҳам сервер ҳангоми
> `POST /sync/sales` аз нав ҳисоб мекунад (клиент танҳо тахминӣ нишон медиҳад).

## 8. Стандартҳои код
- Riverpod providers, бе setState дар мантиқ.
- Model immutable (`freezed`); бе мантиқи бизнес дар widget.
- Хатоҳо → `Failure` → намоиши ягона (snackbar/dialog).
- Папка аз рӯи feature; widget-ҳои такрорӣ дар `shared/`.
- Тест: widget-test барои касса (илова/ҳазфи сатр, ҷамъ).

## 10. Алоқаи Backend ↔ Frontend (хеле муҳим)
- Манбаи ягонаи ҳақиқат = **Swagger/OpenAPI**-и сервер.
- Ҳар тағйири API аввал дар `TZ_01` + Swagger, баъд дар клиент.
- Мантиқи муҳим (FEFO, бақия, нарх) **дар сервер** аст — клиент танҳо намоиш медиҳад.
- Версияи API: `/api/v1` — тағйири breaking → `v2`.
